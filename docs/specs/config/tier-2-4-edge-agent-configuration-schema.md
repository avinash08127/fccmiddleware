## 4. Key Decisions

| Decision | Why | Impact |
|---|---|---|
| Config delivery is a full snapshot, not a patch. | Edge must apply one resolved config version atomically. | `GET /api/v1/agent/config` always returns the complete effective object. |
| Cloud performs the logical push via short polling by the agent. | Existing HLD already defines pull-based config sync. | No websocket or direct portal-to-device channel is required for MVP. |
| Secrets are referenced, not embedded. | FCC and LAN credentials must not be exposed in portal responses or logs. | Config contains `credentialsRef` and `lanApiKeyRef`; secret material is resolved separately on device. |
| Identity fields are immutable after registration. | Changing site or legal entity context changes trust boundaries and audit ownership. | `deviceId`, `siteCode`, and `legalEntityId` require reprovisioning instead of in-place edit. |
| Only listener/session changes require restart. | Polling, buffering, sync cadence, and fiscal rules can change safely at runtime. | Restart-required changes are limited to connection/listener fields; most settings hot-reload. |

### 5.2 Apply Rules

| Rule | Behavior |
|---|---|
| Atomicity | Agent validates the full snapshot first, stores it as staged, then swaps active config only if all validations pass. |
| Provisioning-only fields | Any delta to `agent.deviceId`, `agent.deviceClass`, `site.siteCode`, `site.legalEntityId`, or `api.localApiPort` is rejected and flagged as `REPROVISION_REQUIRED`. |
| Restart-required fields | Agent stores the new version, restarts the foreground service, and applies the new config on startup. No APK reinstall is required. |
| Hot-reload fields | New values take effect in the next in-memory scheduler cycle without service restart. |
| Disconnected sites | When `site.connectivityMode = DISCONNECTED`, agent suspends FCC polling, heartbeat, and local pump-status calls. Existing buffered records remain readable. |
| Last-known-good | If validation fails or compatibility checks fail, the agent keeps the last-known-good active config and emits telemetry/audit for the rejected version. |

### 5.3 Backward Compatibility Handling

| Item | Rule |
|---|---|
| `schemaVersion` | Major version changes are breaking. Agent accepts only matching major versions. Minor versions may add optional fields only. |
| Unknown fields | Agent must ignore unknown fields within the same major schema version. |
| `configVersion` ordering | Incoming config is applied only when `configVersion` is greater than the active version. |
| Version gating | If agent version is below `compatibility.minAgentVersion` or above `compatibility.maxAgentVersion` when set, config is not applied. |
| Recovery path | Rejected configs do not delete the staged or active last-known-good snapshot. Portal and cloud audit must show the rejection reason. |

### 5.4 Cloud-Side Config Management

| Step | Component | Rule |
|---|---|---|
| 1 | Portal | Show `siteCode`, `legalEntityId`, `operatingModel`, and `connectivityMode` as resolved values; master-data-owned fields remain read-only. |
| 2 | Portal | Allow edits only to middleware-owned fields: FCC connection, transaction/ingestion mode, polling, sync, buffer, LAN API, telemetry, and fiscalization override. |
| 3 | Cloud Backend | Validate the submitted payload against `schemas/config/edge-agent-config.schema.json` and cross-field rules before persistence. |
| 4 | Cloud Backend | Persist change set, increment `configVersion`, generate new `configId`, stamp `issuedAtUtc`, create audit entry with actor, before/after diff, and site context. |
| 5 | Cloud Backend | Publish the resolved full snapshot to the config store/cache used by `GET /api/v1/agent/config`. |
| 6 | Edge Agent | Poll `GET /api/v1/agent/config?siteCode={siteCode}&currentVersion={configVersion}` every `sync.configPollIntervalSeconds`. Return `304 Not Modified` when unchanged, otherwise the full snapshot. |
| 7 | Edge Agent | Validate, stage, apply, and report applied or rejected `configVersion` through telemetry on the next telemetry cycle. |

## 6. Validation and Edge Cases
- `api.enableLanApi = true` requires non-null `api.lanApiKeyRef`; otherwise config is invalid.
- `siteHa.enabled = true` requires `siteHa.peerDirectory[]` support in the cloud snapshot even if the array is temporarily empty.
- `siteHa.leaderEpoch` is the fencing token for authoritative writes. Upload, ingest-relay, and pre-auth command APIs must reject stale epochs once HA fencing is active.
- `fccConnection.ingestionMode = CLOUD_DIRECT` still requires `polling.pullIntervalSeconds` because the Edge Agent runs a LAN catch-up poll.
- `fccConnection.transactionMode = PUSH` does not disable polling in `CLOUD_DIRECT`; it only means the FCC's primary upstream path is push.
- `fiscalization.mode = FCC_DIRECT` requires `fiscalization.fiscalReceiptRequired = true`.
- If `site.operatingModel` is `CODO` or `DODO`, customer tax capture rules still come from fiscalization config; operator tax identity remains master-data-owned and is not duplicated in this contract.

## 7. Cross-Component Impact
- Cloud Backend: must expose version-aware `GET /api/v1/agent/config`, validate portal edits, retain audit history, and resolve effective config snapshots.

- Angular Portal: must distinguish read-only master data from middleware-owned editable config and display config-apply status per agent.
