## Overview

### Requirements

#### Infrastructure

The system should be hosted in a cloud infrastructure (server) and accessible via web-interface from any smart device with web-browser. The web-page format should be optimized for smart-phone and normal computer.

#### User interface

The target users of the system would mostly use it on desktop/laptop computers but also on high quality mobile devices. As such the requirement is that the cloud application is mobile friendly and fully usable on a mobile device.

The interface should be light but with very strong visualization capabilities allowing filtering and sorting the results. The highest focus it to be set on visualization of the pumps/station metrics.

#### Data storage

The data from the pump stations is stored in xml-file for each transaction. This xml file is then transmitted over the internet to an ftp-server. The cloud application needs to parse this data and store it in a database in order to aggregate all data and display it in the dashboards described below.

## 1. Transaction management using external host-server

### 1.1 Preconditions

- FDC – FW version 3.49 (or higher)
- The "External Authorization Server" (EAS) is in the same network as FDC
- The IP address of the EAS is allowed by the FDC (see below picture)
- The PORT used for the transaction management is PORT+1 from the defined port at "External Authorization Server" configuration. For example, if the specified port for the external authorization is 5002 then the transaction management port will be 5003.
- Each message contains a SIGNATURE tab. The rules for calculating the signature are described below and are the same as for the "External Authorization"

### 1.2 Calculating the SIGNATURE of the messages

The content closed between the tabs: `<SIGNATURE>` is SHA1 calculation of the content between `<REQ>` `</REQ>` tabs (including them), plus the secret word (password) added immediately after `</REQ>` (with no space in between).

> **Note:** the SHA1 calculation can be verified with the online calculator given below:
> https://md5calc.com/hash/sha1
>
> The value should be exactly the same as with the result generated by the above tool!
> Note that all spaces and special symbols like `/n` matter and will change the signature!

### 1.3 Host communication rules

- For communicating with FDC is used REST API's POST (posting) request from the host.
- The requested device is always supposed to return an answer in order to close the dialog and consider the information as successfully transferred.

### 1.4 Header format of the POST message (from Host to FDC)

Custom Header Keys:

- `Content-Type`: `"Application/xml"`
- `USN-Code`: `"1 – 999999"` (Unique Station Number) – see Note1 below
- `Operation-Code`: `"1"`

**OPERATION CODES:**

| Code | Description |
|------|-------------|
| 1 | TRANSACTION |
| 2 | PRODUCTS and PRICES (explained in the next chapters) |
| 3 | DAY CLOSE (explained in the next chapters) |
| 4 | ATG DATA (explained in the next chapters) |
| 5 | CSR DATA (explained in the next chapters) |

> **Note 1:** The USN-Code should be same as the FDC number given in the system configuration.
>
> **Note 2:** The header should also contain the mandatory key `Content-Length` for message's body length in bytes.

### 1.5 Common parameters of the XML-message transferred between the devices

- `<CMD_CODE>10</CMD_CODE>` - command code (this is a mandatory parameter for each message)

**COMMAND CODES:**

| CMD_CODE | Description |
|----------|-------------|
| 10 | HOST REQUEST FOR TRANSACTION |
| 20 | MODE CHANGE |

- `<CMD_NAME>TRN_REQ</CMD_NAME>` - command name (command description), this tab is optional, it is used for logging and more explanation purposes and can be left empty.

- `<TOKEN>12345</TOKEN>` - a string of max 16 digits (only decimal digits are allowed). It is used to identify that the communication sequence is the same between the host and the FDC. Recommended to have each token with unique number.

**RESPONSE CODES:**

| Code | Description |
|------|-------------|
| 201 | OK / SUCCESS |
| 205 | NO TRANSACTION AVAILABLE |

**ERROR CODES:**

| Code | Description |
|------|-------------|
| 206 | TRANSACTION MODE ERROR |
| 251 | SIGNATURE ERROR |
| 253 | TOKEN ERROR |
| 255 | BAD XML FORMAT |

### 1.6 Messages from Host to FDC

#### MODE CHANGE (20)

Changing the way/mode of how the transactions are transferred from FDC to the host.

**MODES:**

| Mode | Description |
|------|-------------|
| 1 | ON_DEMAND – transfer on host request |
| 2 | UNSOLICITED – unsolicited request from FDC |
| 0 | MODE OFF |

This command is allowed in all modes as long as the other communication rules are fulfilled. The host should receive answer from FDC after a successful transmission of this command!

#### REQUEST FOR TRANSACTION (10)

Requesting the oldest transaction from FDC buffer.

- This command is allowed only in mode "1" - ON_DEMAND. The other modes will ignore it.
- The host should receive answer from FDC after a successful transmission of this command!

### 1.7 Messages from FDC to Host

#### POST A TRANSACTION

Sends the oldest unsent transaction from the buffer in UNSOLICITED mode.

This command is allowed only in mode "UNSOLICITED". The host should respond to FDC after a successful transmission of the transaction.

## 3. External Authorization

### 3.1 Preconditions

- FDC – FW version 3.49 (or higher)
- The "External Authorization Server" (EAS) is in the same network as FDC
- The IP address of the EAS is allowed by the FDC
- DSB mode: POS dependent
- The PORT used for the transaction management is from the defined port at "External Authorization Server" configuration from FDC.
- Each message contains a SIGNATURE tab. The rules for calculating the signature are same as explained in the previous chapters.

### 3.2 Format of the POST message (from Host to Comforter)

**Mandatory Header Keys:**

- `Content-Type`: `"Application/xml"`
- `USN-Code`: `"1 – 999999"` (Unique Station Number) – see Note1 below
- `Operation`: `"Authorize"`

> **Note1:** the USN-Code should be same as the FDC number given in the system configuration.
>
> **Note2:** the header should also contain the standard key `Content-Length` for message's body length in bytes.

**Body:**
```xml
<?xml version="1.0" encoding="utf-8"?>
<FDCMS>
  <AUTH_DATA>
    <PUMP>3</PUMP>
    <FP>1</FP>
    <AUTH>TRUE</AUTH>
    <PROD>2</PROD>
    <PRESET_VOLUME>0.00</PRESET_VOLUME>
    <PRESET_AMOUNT>2000</PRESET_AMOUNT>
    <CUSTNAME>TECHOLOGY Ltd.</CUSTNAME>
    <CUSTIDTYPE>1</CUSTIDTYPE>
    <CUSTID>12345678</CUSTID>
    <MOBILENUM>25588776655</MOBILENUM>
    <DISC_VALUE>10</DISC_VALUE>
    <DISC_TYPE>VALUE</DISC_TYPE>
    <TOKEN>123456</TOKEN>
  </AUTH_DATA>
  <FDCSIGNATURE>484000e79308e7e827359bc90069555115130860</FDCSIGNATURE>
</FDCMS>
```

**Field Descriptions:**

| Field | Description |
|-------|-------------|
| `<PUMP>` | The number of the RDG/DSB in the FDC, assigned to the particular fuel-pump |
| `<FP>` | The filling point number within the current RDG/DSB |
| `<AUTH>` | `TRUE`: activating the authorization, `FALSE`: canceling an active authorization |
| `<PROD>` | Product number to be used, when `0` then all products can be used |
| `<PRESET_VOLUME>` | Preset volume in Liters (or Gallons) |
| `<PRESET_AMOUNT>` | Preset value in the local currency |
| `<CUSTNAME>` | (optional) Customer or company name, if provided it will be assigned to the transaction data |
| `<CUSTIDTYPE>` | (optional) Customer ID type from 1 to 6. If provided it will be assigned to the transaction data |
| `<CUSTID>` | (optional) Customer ID – a unique number identifying the particular customer. If provided it will be assigned to the transaction data |
| `<MOBILENUM>` | (optional) Customer phone number. If provided it will be assigned to the transaction data |
| `<DISC_VALUE>` | (optional) Discount value either in percent or in money per liter. Note: The value should be negative if negative discount is required. Example: `<DISC_VALUE>-10</DISC_VALUE>` |
| `<DISC_TYPE>` | (optional) Type of the discount: `PERCENT` = Percent discount, `VALUE` = Money discount |
| `<TOKEN>` | (optional) Token value from 0 – 65535 that identifies the transaction delivered via the particular preset. This token will be included in the transaction data initiated by the given preset |
| `<FDCSIGNATURE>` | SHA1 calculation of the content between `<AUTH_DATA>` `</AUTH_DATA>` (including) plus a secret word (password) added immediately after `</AUTH_DATA>` with no space in between |

**ID TYPE DESCRIPTION:**

| Type | Description |
|------|-------------|
| 1 | TIN (Tax identification number) |
| 2 | Driving License |
| 3 | Voters Number |
| 4 | Passport |
| 5 | NID (National Identity) |
| 6 | NIL (If there is no ID) |
